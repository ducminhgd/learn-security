## 1. Recon — DH Parameters Matter

### Finite-Field DH Recap

Diffie-Hellman key exchange:

```
Alice chooses secret a, sends A = g^a mod p
Bob   chooses secret b, sends B = g^b mod p
Shared secret: K = B^a = A^b = g^(ab) mod p
```

Security relies on the **Discrete Logarithm Problem**: given `p`, `g`, `A`,
it is computationally infeasible to find `a`. This holds only when:

1. `p` is a large safe prime (2048+ bits for modern standards).
2. The subgroup generated by `g` has large prime order.
3. Both parties validate each other's public key (`A`, `B`) before use.

Break any of these three and DH is attackable.

## 2. Attack 1 — Small Subgroup Confinement

### What It Is

If `p-1` has small prime factors, an attacker can send a public key `A` that
forces the shared secret into a small subgroup. The attacker then learns
the victim's private key modulo the small factor — and if they can observe
enough of these, they recover the full private key via CRT (Pohlig-Hellman).

### Why It Works

The group `ℤ*_p` has order `p-1`. If `p-1 = 2 × 3 × 5 × ...` (many small
factors), the group has subgroups of orders 2, 3, 5, etc.

The attacker sends `A = g^((p-1)/q)` where `q` is a small prime factor. This
element has order `q` in the group. The victim computes:

```
K = A^b mod p = g^(b(p-1)/q) mod p
```

`K` is now one of only `q` possible values. The attacker tries all `q` values
to see which one was used — learning `b mod q`.

Repeat for each small factor and CRT to recover `b`.

### Minimal Exploit

```python
#!/usr/bin/env python3
"""
small_subgroup.py — small subgroup confinement on DH with smooth p-1
"""
from __future__ import annotations

import sympy

def pohlig_hellman_single_factor(g: int, h: int, p: int, q: int) -> int:
    """
    Solve h = g^x (mod p) where x is known to be in range [0, q).
    Uses baby-step giant-step within the subgroup of order q.
    """
    # Baby-step giant-step (BSGS) within subgroup of order q
    m = sympy.isqrt(q) + 1
    # Baby steps: g^j for j = 0..m-1
    baby = {pow(g, j, p): j for j in range(m)}
    # Precompute g^(-m) mod p
    g_inv_m = pow(pow(g, m, p), p - 2, p)
    # Giant steps
    gamma = h
    for i in range(m):
        if gamma in baby:
            return i * m + baby[gamma]
        gamma = (gamma * g_inv_m) % p
    raise ValueError("BSGS failed — x not in expected range")

def small_subgroup_attack(
    p: int,
    g: int,
    victim_public: int,   # B = g^b mod p
    small_factors: list[int],
) -> int:
    """
    Recover victim's private key mod (product of small_factors).
    Returns b mod (p1*p2*...) via CRT.
    """
    residues = []
    moduli   = []

    for q in small_factors:
        # Construct generator of order-q subgroup
        g_q  = pow(g, (p - 1) // q, p)
        # Project victim's key into the subgroup
        h_q  = pow(victim_public, (p - 1) // q, p)
        # Solve DLP in the small subgroup (O(sqrt(q)) BSGS)
        b_q  = pohlig_hellman_single_factor(g_q, h_q, p, q)
        residues.append(b_q)
        moduli.append(q)
        print(f"  [*] b ≡ {b_q} (mod {q})")

    # CRT to combine residues
    from sympy.ntheory.modular import crt
    M, b_partial = crt(moduli, residues)
    return int(b_partial)

# Example: weak p with smooth p-1
# For demo purposes, use a small toy prime
# p-1 = 2^2 × 3 × 5 × 7 × 11 × 13 → p-1 is smooth
from sympy import nextprime

def find_smooth_prime(bits: int, bound: int) -> int | None:
    """Find a prime p such that p-1 is B-smooth (all factors ≤ bound)."""
    # In practice, real weak DH primes are larger; this is for demonstration
    import random
    for _ in range(10000):
        # Construct p-1 as product of primes ≤ bound, then add 1
        n = 1
        primes = list(sympy.primerange(2, bound + 1))
        random.shuffle(primes)
        for pp in primes:
            while n.bit_length() < bits - 2:
                n *= pp
        p_candidate = n + 1
        if sympy.isprime(p_candidate):
            return p_candidate
    return None

# Use a pre-computed smooth-p for the demo
p = 179424673   # Small demo prime; p-1 = 2^5 × 3 × 5 × ... (artificially smooth)
g = 2

# In a real attack, the attacker sends A = g^((p-1)/q) to force
# the victim into a small subgroup. Here we simulate the victim having
# a random private key b.
import random
b = random.randint(2, p - 2)
victim_public = pow(g, b, p)

print(f"[*] p = {p}, g = {g}")
print(f"[*] Victim public key B = {victim_public}")
print(f"[*] Victim private key b = {b} (unknown to attacker)")

# Factorise p-1
factors = sympy.factorint(p - 1)
small_factors = [f for f in factors.keys() if f < 1000]
print(f"[*] Small factors of p-1: {small_factors}")

b_recovered = small_subgroup_attack(p, g, victim_public, small_factors)
print(f"\n[+] Recovered b mod {sympy.Mul(*small_factors)}: {b_recovered}")
# May not be full key recovery for small prime product, but demonstrates the principle
```

## 3. Attack 2 — LOGJAM (CVE-2015-4000)

### What It Is

LOGJAM is a TLS downgrade attack that forced TLS connections to use
512-bit export-grade Diffie-Hellman groups — solvable in hours by an attacker
with precomputed Number Field Sieve (NFS) tables.

### The Attack Chain

```
1. Attacker intercepts the TLS ClientHello (DHE_EXPORT cipher suites listed).
2. Attacker downgrades the ClientHello to request only DHE_EXPORT.
3. Server responds with a 512-bit DH group p.
4. Attacker has pre-computed an NFS factor base for common 512-bit primes.
5. Attacker finishes the discrete log for the session key in real time (~70s).
6. Attacker recovers the shared secret and decrypts/modifies the session.
```

### The Root Cause

Export control laws (until 1999) required US software to use weak cryptography
for export. TLS kept backward compatibility with these "export cipher suites"
for years after the restriction was lifted. LOGJAM exploited this vestigial code.

```
Weak prime: p = 9fdb8b8a004544f0045f1737d0ba2e0b274cdf1a9f588218fb435316a16e374...
               (512 bits — solvable with academic-grade resources)

NFS precomputation: ~1 week of compute for a given p
Per-session computation: ~70 seconds (once the base is precomputed)
```

### Fix

```
TLS configuration:
  - Disable all DHE_EXPORT cipher suites (TLSv1.3 does this by default)
  - Minimum DH group size: 2048 bits
  - Use named safe prime groups (RFC 7919): ffdhe2048, ffdhe3072, ffdhe4096
  - Prefer ECDHE over finite-field DHE (smaller keys, faster, well-standardised)
```

```nginx
# nginx — disable export grades, enforce minimum DH
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:!EXP:!LOW:!NULL';
ssl_dhparam /etc/nginx/dhparam.pem;  # 2048-bit minimum; openssl dhparam -out dhparam.pem 2048
```

## 4. Attack 3 — Invalid Curve Attack (ECDH)

### What It Is

In Elliptic Curve DH (ECDH), both parties operate on points of a specific curve
`y² = x³ + ax + b (mod p)`. If an implementation accepts points from an
**invalid curve** (same `p` but different `a` and `b` values), the attacker
can recover the target's private key.

### Why It Works

```
Normal ECDH:
  Alice sends A = a × G (on valid curve E)
  Bob computes K = b × A = ab × G

Invalid curve attack:
  Attacker sends P = a point on a WEAK curve E' (same field p, different a, b)
  Bob (no point validation) computes K = b × P  ← this is DH on E', not E
  If E' has small prime order q, this is easily solvable
```

The attacker queries Bob with points from progressively smaller-order curves.
Each response leaks `b mod q_i`. CRT recovers the full private key.

### Vulnerable Code vs Fixed Code

```python
# VULNERABLE: no point validation
def vulnerable_ecdh_compute(peer_point: tuple, private_key: int,
                             curve_p: int, curve_a: int) -> int:
    """Accepts any point — no curve membership check."""
    # Scalar multiplication without verifying point is on E
    result = scalar_multiply(private_key, peer_point, curve_a, curve_p)
    return result

# FIXED: validate point is on the curve before use
def fixed_ecdh_compute(peer_point: tuple, private_key: int,
                        curve_p: int, curve_a: int, curve_b: int) -> int:
    """Validates point is on E before computing shared secret."""
    x, y = peer_point
    # Check: y^2 ≡ x^3 + ax + b (mod p)
    lhs = pow(y, 2, curve_p)
    rhs = (pow(x, 3, curve_p) + curve_a * x + curve_b) % curve_p
    if lhs != rhs:
        raise ValueError("Peer public key is not on the curve — attack detected")
    return scalar_multiply(private_key, peer_point, curve_a, curve_p)
```

### Real-World Impact

CVE-2017-10176 (Java JCE invalid curve): Java's ECDH implementation did not
validate that peer public keys were on the specified curve. An attacker could
recover a victim's ECDH private key with as few as 256 queries.

## Key Takeaways

1. DH security is not just about the key size. The prime `p` must be a safe
   prime (or a verifiably random prime with a large-order generator subgroup).
   Smooth `p-1` values allow Pohlig-Hellman in polynomial time.
2. LOGJAM is a historical lesson in backward compatibility. The 20-year tail
   on deprecated export cipher suites created an active attack vector. Regularly
   audit and purge deprecated TLS options — `testssl.sh` makes this trivial.
3. ECDH implementations must validate that peer public keys are points on the
   correct curve. This is one line of code. Omitting it allows full private
   key recovery with ~256 queries. Use a vetted library (OpenSSL, libsodium,
   BoringSSL) rather than rolling custom ECC.
